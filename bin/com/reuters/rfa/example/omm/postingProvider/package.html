StarterProvider_Post is a simple provider application demonstrating how to handle received post messages, send ack messages, and re-send messages received via posts. It can also interactively publish Reuters Domain Model data using the Open Message Model. This application is a server application. One thread dispatches client session connection request, item requests, posts, and timer events. Updates are sent when the timer event is received. StarterProvider_Post provides Logins, Service Directories, Dictionaries, and Market Price. It can be easily extended to provide other types of data.

The StarterProvider_Post supports posting on both Item Stream (On-stream Posting) and Login Stream (Off-stream Posting).

A consumer application, either a Source Distributor or RFA OMMConsumer, must connect to the server to request data, post data, or post messages.

StarterProvider_Post provides examples of how to:
  • Create a Session and an EventQueue
  • Create an OMMProvider event source
  • Listen to client connections
  • Handle request Login, Directory, Dictionary and Items
  • Notify Consumer Application of Posting feature availability through the Login Respone Message
  • Handle Post Messages, Send Ack Messages
  • Re-Send OMM messages received via posts to consumer applications
  • Send Publisher Info with Refresh/Update/Status messages
  • Dispatch events from an EventQueue
  • Cleanup a Session

The StarterProvider_Post application supports the following command line parameters:

  • debug: Enables detailed logging for RFA. The default is false.
  • provSession: Sets the name of the Provider session. The default is myNamespace::provSession.
  • listenerName: Sets a unique name that specifies the connection on which to listen. The default is ""(null).
  • acceptSession: Accepts or denies a Session. The default is true.
  • runTime: Sets the duration (in seconds) for which the application runs before exiting. The default is 600.
  • rdmFieldDictionary: Sets the RDM Field dictionary name and location. The default is /var/triarch/RDMFieldDictionary.
  • enumType: Sets the enum dictionary name and location. The default is /var/triarch/enumtype.def.
  • serviceName: Sets the name of the service for the SrcDirectory response. The default is DIRECT_FEED.
  • sendUpdates: Turns on/off item updates. The default is false.
  • updateInterval: Sets the update interval in seconds. The default is 1.
  • updateRate: Sets the update rate per interval. The default is 2.
  • supportPost: Sets whether to support OMM Post feature. The default is true.
  • ackList: Comma-delimited list of post ids that require ack messages. This parameter supports a range of numbers. The Default is 1-1000,1500.
  • positiveAck: Sets the application to send Positive ack or Negative ack messages. The Default is true.
  • setStatusCode: Sets the status code in the ack. This parameter is not required for positive acks. The default is false.
  • setStatusText: Set Status text in the ack. This parameter is not required for positive acks. The default is false.
  • setStateUsingOMMState: Sets whether to include the ack Status indirectly on the OMMMsg using OMMState. The default is true.
  • ackEncodeAttrib: Sets whether to encode Attrib data in the ack Message. The default is false.
  • ackEncodePayload: Sets whether to encode Payload data in the ack Message. The default is false.
  • sendPublisherInfo: Sets the application to send Publisher Info on Refresh/Update/Status messages. The default is true.
  • forwardPostPayloadMsg: Sets the application to send post payload OMMMsg to the consumer application. Only item streams are supported. The default is true.

You can run the example as an application with the following commands:

    com.reuters.rfa.example.omm.postingProvider.StarterProvider_Post -provSession rsslJava::aProviderLocal02  -sendUpdates false -updateInterval 1 -updateRate 2 -supportPost true -ackList 1-2,4 -positiveAck true -setStatusCode false -setStatusText false -setStateUsingOMMState true -ackEncodeAttrib false -ackEncodePayload false -sendPublisherInfo true -forwardPostPayloadMsg true
