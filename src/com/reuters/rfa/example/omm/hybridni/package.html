The following section contains a hybrid application that is both an OMM consumer and a non-interactive OMM provider.

The {@link com.reuters.rfa.example.omm.hybridni.HybridNIDemo HybridNIDemo} demonstrates how to write an application that is both an OMM consumer and a non-interactive OMM provider. The OMM provider side {@link com.reuters.rfa.session.omm.OMMProvider OMMProvider} of HybridNIDemo logs into the source distributor. The OMM consumer side {@link com.reuters.rfa.session.omm.OMMConsumer OMMConsumer} logs into a provider application, requests directory/items, and publishes them to the source distributor.

HybridNIDemo supports the following command line arguments:

  • runTime: Sets the number of seconds that application will run. The default is 600.
  • session: The Session must contain at least two connections: one RSSL and the other RSSL_NIPROV. The default is myNamespace::hybridNISession.
  • itemName: Sets the list of items to open separated by ','. The default is TRI.N.
  • mmt: Sets the Message Model Type. The default is MARKET_PRICE.
  • serviceName: Sets the service used for requests. The default is DIRECT_FEED.
  • attribInfoInUpdates: Sets the provider application to send OMMAttribInfo in its update/status messages. The default is false.
  • user: Sets the DACS username for Login. The default is the value of the user.name system property.
  • position: Sets the DACS position for Login. The default is IP address/hostname.
  • application: Sets the DACS application ID for Login. The default is 256.

You can run the HybridNIDemo as an application with the following command:

    java com.reuters.rfa.example.omm.hybridni.HybridNIDemo -runTime 50 -session myNamespace::localOMMHybridNI -serviceName DIRECT_FEED -itemName IBM.N,MSFT.O -user "username"

Assumptions:

  • The provider application, HybridNIDemo, and the source distributor must use the same service name.
  • You must limit the number of RSSL_NIPROV connections configured in the session to one.
